# AI Agent System: Подробное описание реализации

**AI Agent System** - это современная система на базе Laravel и AI SDK, построенная по принципу "мульти-агентности" (Multi-Agent System). Она позволяет решать сложные задачи, разбивая их на мелкие шаги, выбирая нужные инструменты и "обдумывая" промежуточные результаты.

---

## 1. Общая архитектура системы

Система построена на **событийно-ориентированном подходе (Event-Driven Architecture)**. Это значит, что вместо одного длинного и жесткого программного кода, выполнение задачи происходит через цепочку событий.

### Это дает следующие преимущества:
- **Гибкость**: Вы можете легко добавить новое действие (например, отправку письма или логирование в Telegram), просто подписавшись на нужное событие.
- **Прозрачность**: На каждом этапе мы точно знаем, что делает агент: "планирует", "ищет информацию", "считает" или "отвечает".

### Ключевые события (Events):
- **`UserMessageReceived`**: Точка входа. Пользователь отправил сообщение.
- **`PlanCreated`**: Система составила план действий.
- **`ToolCalled`**: Вызван конкретный инструмент (например, калькулятор).
- **`ReflectionGenerated`**: Агент проанализировал результат и решил, что делать дальше.
- **`StepCompleted`**: Текущий шаг задачи завершен.

---

## 2. Основные компоненты (Мозги системы)

### Supervisor (Супервизор) — Главный оркестратор
Супервизор — это "начальник", который принимает запрос от пользователя и управляет циклом выполнения задачи. В отличие от жестких планов, Супервизор использует динамическое планирование.
- **Класс**: `App\Ai\Core\Supervisor`
- **Метод**: `handle(string $userMessage)` — точка входа, запускает динамический цикл.

### Dynamic Planner — Динамический планировщик
Планировщик решает, какой шаг будет следующим, анализируя историю уже выполненных действий и текущие результаты.
- **Класс**: `App\Ai\Core\DynamicPlanner`
- **Интерфейс**: `App\Ai\Core\Interfaces\DynamicPlannerInterface`
- **Методы**:
    - `initialStep(string $message)` — определяет самый первый шаг (обычно `research`).
    - `nextStep(AgentState $state)` — анализирует состояние и возвращает следующий `OrchestrationStep` или `null` для завершения.

### Orchestration Executor — Исполнитель шагов
Этот компонент отвечает за запуск конкретного агента на текущем шаге.
- **Класс**: `App\Ai\Core\OrchestrationExecutor`
- **Метод**: `runStep(AgentState $state)` — выполняет текущий шаг, обновляет историю и контекст в стейте.

## 3. Динамическая модель выполнения (Dynamic Plan) 🔥

В системе реализован подход **Dynamic Plan** (аналогично NeuronAI / LangGraph). План не задан заранее, он строится "на лету":

1. **Инициализация**: `Supervisor` получает запрос и запрашивает первый шаг у `DynamicPlanner`.
2. **Цикл выполнения**:
    - `Executor` запускает назначенного агента (`ResearchAgent` или `SummaryAgent`).
    - Агент выполняет работу и записывает результат в историю и контекст `AgentState`.
    - `DynamicPlanner` анализирует обновленный `AgentState` и историю.
    - Если нужны уточнения — планировщик назначает новый шаг.
    - Если данных достаточно — планировщик назначает `SummaryAgent` или завершает процесс.
3. **Завершение**: Генерируется событие `WorkflowCompleted`, когда планировщик возвращает `null`.

### Преимущества динамического подхода:
1. **Глубокий контекст**: Агенты видят историю предыдущих шагов и не дублируют работу.
2. **Адаптивность**: Система может сама решить провести дополнительное исследование, если первичных данных оказалось мало.
3. **Прозрачность**: Каждое решение планировщика сопровождается объяснением (`thought`).
4. **Легкость**: Отсутствие сложной структуры плана, только текущий шаг и история.

## 4. Специализированные агенты

В системе есть несколько типов агентов, каждый из которых решает определенные задачи:

### ResearchAgent (Исследователь)
Умеет искать информацию в базе знаний и выполнять расчеты. Он работает внутри цикла (Loop), пока не найдет ответ.
- **Класс**: `App\Ai\Agents\ResearchAgent`
- **Особенность**: Использует `LoopController` для итеративного поиска.

### SummaryAgent (Саммаризатор)
Специализируется на сжатии и структурировании информации. Получает "кучу данных" и превращает их в краткий отчет.
- **Класс**: `App\Ai\Agents\SummaryAgent`

## 4. Инструменты и База знаний (RAG)

Агенты не просто "галлюцинируют", они пользуются реальными инструментами:

### Реестр инструментов (Tool Registry)
Позволяет динамически добавлять новые навыки агентам.
- **Класс**: `App\Ai\Core\ToolRegistry`
- **Методы**: `register()` — добавляет инструмент; `getTools()` — отдает список для ИИ.

### Реализованные инструменты:
- **`CalculatorTool`**: Для точных математических вычислений.
- **`VectorSearchTool`**: Для поиска по загруженным документам.

### Поиск по знаниям (RAG - Retrieval-Augmented Generation)
Система векторизует (превращает в числа) тексты документов и сохраняет их в базу данных PostgreSQL с расширением `pgvector`. Когда агент ищет информацию, он сравнивает "смысл" запроса со "смыслом" фрагментов текста в базе.
- **Класс**: `App\Ai\Memory\VectorStore`
- **Метод**: `search()` — выполняет семантический поиск.

## 5. Алгоритм работы (Жизненный цикл запроса)

Когда пользователь отправляет запрос, система проходит через следующие этапы:

### Этап 1: Инициализация
1. **Точка входа**: Вызывается метод `Supervisor::handle(string $userMessage)`.
2. **Первый шаг**: `Supervisor` запрашивает начальное действие у `DynamicPlanner::initialStep()`.
3. **Создание состояния**: Инициализируется `AgentState` с исходным запросом и первым шагом.
4. **Событие**: Генерируется событие `WorkflowStarted`.

### Этап 2: Цикл выполнения (Dynamic Loop)
В методе `Supervisor::runCycle()` происходит итеративный процесс:

1. **Выполнение шага**: `Supervisor` вызывает `OrchestrationExecutor::runStep($state)`.
2. **Работа агента**:
   - `Executor` находит нужного агента и вызывает его `execute()`.
   - Агент (`ResearchAgent` или `SummaryAgent`) выполняет задачу, используя историю предыдущих шагов.
   - Результат сохраняется в `history` и `context` объекта `AgentState`.
3. **Планирование следующего шага**:
   - `Supervisor` вызывает `DynamicPlanner::nextStep($state)`.
   - Планировщик анализирует историю и решает: нужно ли уточнение (`research`), финальный вывод (`summary`) или завершение (`null`).
4. **Обновление состояния**: Если возвращен новый шаг, он устанавливается в `state->step`, и цикл повторяется.

### Этап 3: Завершение
1. **Выход из цикла**: Когда планировщик возвращает `null`, цикл прерывается.
2. **Событие**: Генерируется событие `WorkflowCompleted`.
3. **Результат**: `Supervisor::handle()` возвращает финальный `AgentState` с итоговым ответом в поле `context`.

## 6. Технические особенности для разработчиков

- **Reasoning Stream (Ход мыслей)**: Вы можете наблюдать за процессом работы в реальном времени через веб-интерфейс (Server-Sent Events). Каждое событие (вызов инструмента, решение рефлектора) мгновенно отображается на фронтенде.
- **Guardrails (Безопасность)**: В `OrchestrationExecutor` встроены проверки. Если ИИ предложит вызвать несуществующего агента, система заблокирует этот шаг.
- **Docker и Окружение**:
  - PHP 8.4-cli.
  - PostgreSQL + pgvector для памяти.
  - Redis для очередей и событий.
  - Ollama для локального запуска LLM моделей (например, `llama3`, `mistral`).
- **Тестирование**: Система покрыта тестами (`MultiAgentTest.php`, `LoopControllerTest.php`), которые проверяют как отдельных агентов, так и их совместную работу.

## 7. Динамическое планирование (Dynamic Plan) 🔥

В системе реализован подход **Dynamic Plan** (аналогично NeuronAI / LangGraph). В отличие от статического планирования, план строится "на лету" после каждого выполненного шага.

### Ключевые особенности:
1. **Глубокий контекст**: Агенты (`ResearchAgent`, `SummaryAgent`) получают полный объект `AgentState`, что позволяет им видеть историю предыдущих шагов и не дублировать работу.
2. **Итеративное уточнение**: `DynamicPlanner` может запускать одного и того же агента несколько раз с разными задачами, если первичных данных недостаточно.
3. **Синтез знаний**: `SummaryAgent` собирает данные из всех шагов `research` в истории, а не только из последнего результата.
4. **Прозрачность**: Каждое решение планировщика сопровождается полем `thought`, объясняющим логику выбора следующего шага.

Этот подход делает систему гораздо более адаптивной к сложным и неопределенным задачам.
