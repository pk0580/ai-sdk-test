# Описание реализации AI Agent System

## Шаг 1: Event-driven ядро

В качестве первого шага реализована инфраструктура событий (Laravel Events) для управления жизненным циклом агента.

### Почему события?
Использование событий позволяет:
- Избавиться от жестко закодированных циклов `while`.
- Сделать систему расширяемой (можно добавить логирование, уведомления или новых агентов без изменения ядра).
- Реализовать асинхронную обработку шагов через очереди.

### Основные события:
- `UserMessageReceived`: Входная точка, сообщение от пользователя.
- `PlanCreated`: Создан план действий (массив шагов).
- `ToolCalled`: Вызван конкретный инструмент (Tool).
- `ToolResultReceived`: Получен результат работы инструмента.
- `ReflectionGenerated`: Агент проанализировал результат и решил, что делать дальше.
- `StepCompleted`: Текущий шаг плана завершен.

## Шаг 3: Tool Registry

На этом этапе реализован реестр инструментов (`ToolRegistry`), который служит связующим звеном между планировщиком и исполнителями действий.

### Что делает Реестр?
1. **Регистрация**: Позволяет динамически добавлять инструменты (например, `calculator`, `vector_search`) через `AppServiceProvider`.
2. **Метаданные**: Автоматически извлекает описание и JSON-схемы параметров инструментов, используя контракты Laravel AI SDK (`Laravel\Ai\Contracts\Tool`).
3. **Обнаружение**: Предоставляет Планировщику список доступных инструментов в формате, который понимает LLM.

### Почему это важно?
- **Типизация**: Каждый инструмент имеет четкую схему данных (`schema`), что минимизирует ошибки при передаче параметров от ИИ.
- **Расширяемость**: Добавление нового навыка агенту сводится к созданию одного класса и его регистрации одной строкой кода.
- **Интеграция с SDK**: Используются стандартные интерфейсы Laravel AI SDK, что делает код чистым и совместимым с экосистемой.

### Реализованные инструменты:
1. **CalculatorTool**: Выполняет математические вычисления.
2. **VectorSearchTool**: Ищет информацию в базе знаний (Шаг 4).

## Шаг 4: RAG (Retrieval-Augmented Generation)

Реализована система поиска по векторизованной базе знаний, позволяющая агенту отвечать на вопросы, основываясь на загруженных документах.

### Компоненты системы:
1. **База данных (PostgreSQL + pgvector)**: Используется расширение `vector` для хранения и эффективного поиска семантически похожих текстов.
2. **Модель Document**: Хранит контент и его эмбеддинг (векторное представление из 768 измерений).
3. **VectorStore**: Сервис для управления документами:
   - Генерирует эмбеддинги через Laravel AI SDK (модель `nomic-embed-text` в Ollama).
   - Выполняет поиск ближайших соседей (Nearest Neighbors) по косинусному сходству или L2-расстоянию.
4. **VectorSearchTool**: Обертка над хранилищем, которую использует AI-агент во время планирования.

### Процесс (RAG Flow):
1. **Индексация**:
   - Текст документа разбивается на чанки (параграфы) с 25% перекрытием (overlap) для сохранения контекста.
   - Каждому чанку присваивается `document_id` и `chunk_index` в метаданных.
   - Чанки превращаются в векторы (embeddings) и сохраняются в БД.
2. **Поиск**:
   - Поисковый запрос векторизуется.
   - Выполняется поиск ближайших соседей.
3. **Дедупликация**:
   - Результаты группируются по `document_id`.
   - Из каждой группы берется топ N чанков (по умолчанию 2), чтобы избежать перенасыщения контекста фрагментами из одного и того же документа.
4. **Использование**: Найденные фрагменты передаются ИИ-агенту.

### Технические детали:
- **Размерность вектора**: 768 (соответствует модели `nomic-embed-text`).
- **Инструменты**: Пакет `pgvector/pgvector-laravel` для работы с векторами в Eloquent.
- **Тесты**: `RagTest.php` покрывает создание документов с чанкованием, дедупликацию в поиске, генерацию эмбеддингов и работу инструмента.

### Технические детали (инфраструктура)
1. **Docker Окружение**:
   - Приложение работает на PHP 8.4-cli (официальный образ).
   - Основные сервисы: `app` (PHP), `db` (PostgreSQL + pgvector), `redis`, `worker`, `horizon`, `scheduler`, `ollama`.
   - Запуск тестов в докере: `php artisan test`.

2. **Event-driven Цикл**:
   - Команда `ai:test` инициирует цепочку: `UserMessageReceived` -> `PlanCreated` -> `ToolCalled` -> `ToolResultReceived` -> `ReflectionGenerated` -> `StepCompleted` / `Finish`.
   - Текущая реализация эмулирует выполнение инструментов и принятие решений (Reflection), логируя каждый этап в `laravel.log`.

3. **Логирование**:
   - Все события логируются в `storage/logs/laravel.log`, что позволяет отслеживать "мысли" агента.

## Шаг 2: Planner (Планировщик)

На этом этапе реализован мозг агента — Планировщик (`Planner`), который использует LLM для превращения запроса пользователя в конкретный план действий.

### Что делает Планировщик?
1. **Анализирует запрос**: Например, "Сколько будет 25 * 17?".
2. **Выбирает инструменты**: Для расчетов выбирается `calculator`, для поиска знаний — `vector_search`.
3. **Формирует шаги**: Создает последовательность действий в формате JSON, понятном для программного выполнения.

### Почему это важно?
- **Экономия ресурсов**: Агент сразу видит путь к решению, что уменьшает лишние обращения к ИИ.
- **Предсказуемость**: Мы получаем структурированный список шагов (DTO `Plan` и `Step`) вместо нечеткого текста.
- **Обработка ошибок**: Если ИИ ошибается или не отвечает, предусмотрен "запасной" план (fallback).

### Технические детали
1. **Класс Planner**:
   - Использует `Laravel\Ai\AnonymousAgent` для выполнения запросов.
   - Имеет жестко заданный системный промпт с описанием доступных инструментов из `ToolRegistry`.
   - Ответы LLM парсятся для извлечения JSON.

2. **Интеграция**:
   - Планировщик вызывается на первом этапе цикла в `LoopController`.
   - Результат упаковывается в DTO `Plan`, содержащий коллекцию `Step`.

## Шаг 5: Reflection (Рефлексия)

На этом этапе реализован механизм "самоанализа" агента — Рефлектор (`Reflector`), который позволяет ИИ оценивать результаты выполнения каждого шага и принимать решение о дальнейших действиях.

### Что делает Рефлектор?
1. **Анализирует контекст**: Сопоставляет исходный запрос пользователя, выполненный шаг и полученный результат от инструмента.
2. **Принимает решение**: Определяет, достаточно ли полученной информации для финального ответа (`finish`) или требуется продолжение работы (`continue`).
3. **Предлагает уточнения**: Если данных недостаточно, Рефлектор может предложить следующий шаг или уточнить параметры для повторного вызова инструмента.

### Почему это важно?
- **Автономность**: Агент может самостоятельно исправлять ошибки (например, если поиск не дал результатов, попробовать другой поисковый запрос).
- **Точность**: Рефлексия гарантирует, что ответ будет полным и релевантным запросу пользователя, а не просто "первым попавшимся" результатом.
- **Управляемость**: Вместо бесконечного цикла мы получаем осознанный процесс принятия решений.

### Технические детали
- **Формат ответа**: LLM возвращает структурированный JSON с полями `decision`, `thought` и `next_suggestion`.
- **Интеграция**: Вызывается внутри `LoopController` после каждого выполнения инструмента.
- **Логирование**: Все "мысли" рефлектора сохраняются в логи для последующего дебага.

## Шаг 7: Multi-agent System

На этом этапе реализована архитектура с несколькими специализированными агентами под управлением Супервизора.

### Компоненты системы:
1. **Supervisor (Супервизор)**: 
   - Выступает в роли диспетчера запросов.
   - Анализирует задачу пользователя и решает, какого агента вызвать или какую цепочку (chain) агентов построить.
   - Использует комбинацию эвристик и LLM для принятия решений.
2. **ResearchAgent (Исследователь)**:
   - Специализируется на поиске информации и выполнении расчетов.
   - Использует `LoopController` для итеративного решения задач с помощью инструментов (`CalculatorTool`, `VectorSearchTool`).
3. **SummaryAgent (Саммаризатор)**:
   - Специализируется на синтезе и сжатии информации.
   - Получает на вход массив данных и превращает его в краткий структурированный отчет.

### Алгоритм работы (Multi-agent Flow):
1. **Прием запроса**: Супервизор получает сообщение пользователя.
2. **Оркестрация**: 
   - Если задача простая (например, только расчет), Супервизор вызывает `ResearchAgent`.
   - Если задача сложная (например, "найди и резюмируй"), Супервизор строит цепочку: `ResearchAgent` -> `SummaryAgent`.
3. **Выполнение**: Агенты последовательно выполняют свои задачи. Результат одного агента передается как входные данные для следующего в цепочке.
4. **Финальный ответ**: Итоговый результат возвращается пользователю.

### Технические детали:
- **Интерфейсы SDK**: Все агенты наследуются от `BaseAgent` и имплементируют контракты Laravel AI SDK: `Agent`, `Conversational`, `HasTools`, `HasStructuredOutput`. Это позволяет использовать встроенные механизмы фейков (`Ai::fakeAgent`) и логирования.
- **Интеграция**: `ResearchAgent` интегрирован с `LoopController`, что позволяет ему использовать всю мощь событийного ядра и инструментов.
- **Тесты**: `MultiAgentTest.php` проверяет корректность выбора агентов Супервизором и корректную передачу данных в цепочке.

## Шаг 6: Loop Controller (Цикл управления)

`LoopController` — это центральный управляющий компонент, который объединяет Планировщик и Рефлектор в единый итеративный цикл решения задачи.

### Как работает Цикл?
1. **Начало**: Принимает сообщение пользователя и запрашивает у `Planner` первоначальный список шагов.
2. **Выполнение**: Последовательно берет шаги из очереди и выполняет соответствующие инструменты через `ToolRegistry`.
3. **Анализ**: Передает результат каждого шага в `Reflector`.
4. **Динамика**:
   - Если Рефлектор говорит `finish`, цикл прерывается и формируется итоговый ответ.
   - Если Рефлектор говорит `continue` и предлагает новый шаг, этот шаг добавляется в начало очереди выполнения.
5. **Защита**: Ограничение по максимальному количеству итераций (по умолчанию 5) предотвращает бесконечные циклы в случае галлюцинаций или ошибок LLM.

### Результат работы:
В конце выполнения `LoopController` формирует структурированный отчет, включающий:
- Финальный анализ ситуации.
- Пошаговую историю выполненных действий с результатами каждого инструмента.

### Архитектурные изменения (Рефакторинг)
- **Тестирование**: Добавлен `LoopControllerTest`, который проверяет как линейное выполнение плана, так и многошаговые сценарии с дозаказом новых действий через рефлексию. Все тесты переведены на использование `AnonymousAgent::fake()` для стабильности.
- **Service Provider**: Реестр инструментов и зависимости ядра регистрируются в `AppServiceProvider`.
